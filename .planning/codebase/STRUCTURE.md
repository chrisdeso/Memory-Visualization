## Directory Purposes

**`web/`:**
- Purpose: Production web visualization application served to users
- Contains: HTML markup, CSS styling, JavaScript visualization modules
- Key files: `index.html` (entry point), `js/*.js` (visualization logic)

**`web/js/`:**
- Purpose: Modular JavaScript visualization components
- Contains: D3.js-based visualizations for memory regions, timeline control logic
- Key files:
  - `visualization.js` - Main orchestrator (newer pattern)
  - `controller.js` - Alternative orchestrator (legacy pattern)
  - `{heap,stack,statics}-visualization.js` - Memory region renderers
  - `timeline-*.js` - Navigation components (two versions)

**`demo/`:**
- Purpose: Standalone demo/reference implementation
- Contains: Self-contained HTML with full visualization (largely duplicates `web/index.html`)
- Key files: `index.html` (2.3KB, embedded all CSS/JS inline)

**`docs/`:**
- Purpose: Project documentation and design artifacts
- Contains: Architecture diagrams, workflow guides, proposal documents
- Key files: `architecture.md` (Mermaid diagram of full system including missing C++ backend)

## Key File Locations

**Entry Points:**
- `web/index.html`: Primary web application served by HTTP server
- `demo/index.html`: Self-contained standalone demo (for offline testing)
- Both expect `trace.json` to be present in same directory at runtime

**Configuration:**
- No configuration files (hardcoded values for colors, dimensions, margins)
- Runtime data: `trace.json` (generated by backend, not in repo)

**Core Logic:**
- `web/js/visualization.js`: Main orchestrator class, loads trace, coordinates visualizations
- `web/js/controller.js`: Alternative orchestrator, simpler approach with global variables
- `web/js/{heap,stack,statics}-visualization.js`: Memory region rendering classes

**Timeline Navigation:**
- `web/js/timeline-navigation.js`: D3-based slider with play/pause controls, drag support
- `web/js/timeline-nav.js`: D3 timeline with point visualization

**Styling:**
- Inline CSS in `web/index.html` and `demo/index.html` (no separate stylesheets)
- Color scheme: Teal header (#00b8a9), pink stack, orange static, blue heap

## Naming Conventions

**Files:**
- Kebab-case: `timeline-navigation.js`, `heap-visualization.js`
- HTML entry points: `index.html`

**Directories:**
- Lowercase: `web/`, `demo/`, `docs/`
- No version numbers or timestamps

**JavaScript Classes:**
- PascalCase: `MemoryVisualization`, `HeapVisualization`, `StackVisualization`, `TimelineNavigation`

**CSS Classes:**
- Kebab-case: `memory-region`, `memory-table`, `stack-header`, `code-line`, `highlight-row`

**JavaScript Variables & Functions:**
- camelCase: `currentStep`, `traceData`, `renderStep()`, `loadTrace()`, `updateDisplay()`
- Global: `steps`, `currentStep` (in controller.js), `sourceCode`, `traceData` (in index.html)

## Where to Add New Code

**New Memory Region Type (e.g., Stack vs Heap vs Static variations):**
- Implementation: Create new file `web/js/{region-type}-visualization.js`
- Pattern: Copy structure from `web/js/stack-visualization.js`
  - Class constructor takes `containerId` selector
  - Implement `initTable()` to set up DOM structure
  - Implement `update(data)` to re-render with new state
  - Use D3.js to manipulate DOM if advanced interactions needed
- Instantiate in `web/js/visualization.js` constructor and add to `updateVisualizations()` method

**New Navigation Component:**
- Implementation: Create `web/js/{feature}-navigation.js`
- Pattern: Similar to `TimelineNavigation` - manages state and triggers callbacks
- Hook into `updateVisualizations()` via callback pattern (e.g., `onTimeChange` callback)

**New Data Visualization Feature (e.g., memory leak highlighting):**
- Pattern: Add method to existing visualization class (e.g., `HeapVisualization.getBlockColor()`)
- Test: Modify trace.json fixture to include new data fields, verify rendering

**Backend Integration (C++ Memory Tracer):**
- Purpose: Generate trace.json from user program
- Output format: `{ steps: [{ line_number, description, stack: [...], heap: {...}, static: [...] }, ...] }`
- Expected structure per step:
  - `line_number`: Current line being executed
  - `description`: Human-readable explanation of this step
  - `stack`: Array of stack frames with function names and variables
  - `heap`: Object with `memory_blocks` array of allocated/deallocated blocks
  - `static`: Array of global/static variables

## File Dependencies & Module Organization

**`web/index.html`:**
- Dependencies: None (self-contained with inline CSS and script)
- Consumed by: Web browsers
- Data source: `trace.json` (fetched at runtime)

**`web/js/visualization.js` pattern:**
- Dependencies: `HeapVisualization`, `StackVisualization`, `StaticsVisualization`, D3.js library
- Consumed by: `web/index.html` (if using this pattern)
- Loads: `trace.json` via fetch

**`web/js/controller.js` pattern:**
- Dependencies: D3.js library (imported in HTML via script tag)
- Consumed by: `web/index.html` (if using this pattern)
- Loads: `trace.json` via fetch

**Memory Visualization Classes** (`heap-visualization.js`, etc.):
- Dependencies: D3.js library
- Consumed by: `visualization.js` or `index.html` inline script
- Data source: Passed via `update(data)` method

**Timeline Components**:
- Dependencies: D3.js library
- Consumed by: Alternative orchestration patterns
- Callbacks: `onTimeChange` callback to signal navigation
